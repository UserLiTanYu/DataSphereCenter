# DataSphereCenter 项目阶段提示词合集

## 说明

本文档根据用户前期真实对话内容优化整理而成。

目标是把每一段原始需求，改写成更清晰、更专业、更适合直接交给 AI 执行的提示词。

项目中文名：数境中枢

项目英文名：DataSphereCenter

## 1. 项目命名提示词

### 原始需求

```text
你是一个数据大屏专家，我是一个小白，需要开发一个数据大屏项目，现在需要一个名字，请你给我8个中文名和英文名，并给我推荐一个，使用大驼峰命名法
```

### 优化后提示词

```text
你是一名资深数据大屏产品命名专家和前端项目顾问。

我是前端初学者，准备开发一个数据可视化大屏项目，目前需要为项目确定一个专业、好记、适合开源仓库和前端工程使用的名称。

请你为我提供 8 组项目名称，每组都包含：

1. 中文名
2. 英文名
3. 简短命名含义
4. 适合的使用场景

英文名必须使用大驼峰命名法，例如 DataVisionCenter。

命名风格要求：

1. 适合数据大屏、数据可视化、运营监控、智慧中枢等场景。
2. 中文名要专业、有科技感，但不要太生僻。
3. 英文名要适合作为项目名、仓库名、前端工程名。
4. 不要使用过长、难读、难拼写的英文名。

最后请从 8 组名称中推荐你认为最适合的一组，并说明推荐理由。
```

## 4. GitHub 推送提示词

### 原始需求

```text
git@github.com:UserLiTanYu/DataSphereCenter.git，帮我提交并推送
```

### 优化后提示词

```text
请将当前本地 Git 仓库提交并推送到 GitHub。

远程仓库地址：

git@github.com:UserLiTanYu/DataSphereCenter.git

请直接执行以下任务：

1. 检查当前 Git 工作区状态。
2. 如果存在未提交内容，请先查看变更范围。
3. 将需要提交的项目文件加入暂存区。
4. 创建一次语义清晰的 Git 提交。
5. 如果当前分支不是 main，请将主分支调整为 main。
6. 配置远程仓库 origin。
7. 将 main 分支推送到 GitHub。
8. 设置本地 main 分支跟踪 origin/main。

注意事项：

1. 不要删除用户已有文件。
2. 不要执行 git reset --hard 等危险操作。
3. 如果远程仓库已存在 origin，请先检查地址是否一致。
4. 如果 SSH 权限或 known_hosts 出现问题，请说明原因并继续尝试安全解决。

完成后请告诉我：

1. 推送是否成功。
2. 当前分支名称。
3. 远程仓库地址。
4. 最新提交信息。
5. 当前工作区是否干净。
```

## 5. 技术架构方案提示词

### 原始需求

```text
请你给给出你认为最好的三种技术架构方案。这个项目目前是纯前端，数据采用mock,后期需要兼容api切换，必须考虑测试系统，日志系统和代码质量系统，必须遵循模块化开发的规范，不能把所有的代码堆在一个文件里
```

### 优化后提示词

```text
你是一名资深数据大屏前端架构师。

我正在开发一个数据可视化大屏项目，目前项目处于纯前端阶段，数据暂时使用 Mock 数据。后期项目需要平滑切换到真实 API。

请你基于这个背景，给我设计 3 套你认为最适合的前端技术架构方案。

项目要求：

1. 当前阶段是纯前端项目。
2. 当前数据来源使用 Mock。
3. 后期必须兼容真实 API 切换。
4. 必须考虑测试系统。
5. 必须考虑日志系统。
6. 必须考虑代码质量系统。
7. 必须遵循模块化开发规范。
8. 不能把所有代码堆在一个文件里。
9. 项目要适合前端初学者继续学习和维护。
10. 项目类型是数据可视化大屏，不是普通后台管理系统。

请每套方案都包含：

1. 技术栈组成
2. 推荐目录结构
3. Mock/API 切换设计
4. 测试系统设计
5. 日志系统设计
6. 代码质量系统设计
7. 方案优点
8. 方案缺点
9. 适合什么阶段或什么类型的项目

最后请你从 3 套方案中推荐最适合我的一套，并说明原因。

请用通俗易懂的方式解释，我是前端初学者。
```

## 6. 完整项目开发提示词

### 原始需求

```text
你是一个数据大屏专家。项目的英文名是DataSphereCenter ，中文名是数境中枢。请你根据你刚才推荐的方案，组织一份优秀的、符合规范的、完整的提示词给我。我希望把提示词给 AI 后，它就能按照计划执行，最终自动打开浏览器，让我看到一个漂亮的数据可视化大屏页面。
```

### 优化后提示词

```text
你是一名资深数据大屏前端架构师和工程化专家。

我要开发一个数据可视化大屏项目。

项目英文名：DataSphereCenter
项目中文名：数境中枢

请你基于 Vue3 + Vite + TypeScript 技术方案，完整搭建一个可运行、可维护、模块化的数据大屏前端项目。

请不要只输出方案，请直接在当前项目目录中创建和修改文件、安装依赖、执行检查、启动服务，并自动打开浏览器，让我看到一个漂亮的数据可视化大屏页面。

一、技术架构要求

请使用以下技术栈：

1. Vue 3
2. Vite
3. TypeScript
4. Pinia
5. Vue Router
6. ECharts
7. Axios
8. Vitest
9. Playwright
10. ESLint
11. Prettier
12. Stylelint

项目当前阶段使用纯前端 Mock 数据，后期必须支持真实 API 切换。

二、核心目标

请完成以下任务：

1. 初始化 Vue3 + Vite + TypeScript 项目。
2. 建立规范的模块化目录结构。
3. 配置路由、状态管理、图表能力、请求工具和 Mock 数据层。
4. 实现 Mock/API 可切换的数据服务层。
5. 实现一个完整的数据可视化大屏首页。
6. 配置测试系统，包括单元测试和端到端测试。
7. 配置日志系统。
8. 配置代码质量系统。
9. 更新 README 文档。
10. 安装依赖并完成验收检查。
11. 启动本地开发服务器。
12. 自动打开浏览器展示最终页面。

三、目录结构要求

请按照以下结构组织代码，禁止把所有代码堆在一个文件里：

src/
├─ app/                 # 应用初始化、插件注册、路由配置
├─ assets/              # 图片、字体等静态资源
├─ components/          # 通用组件
├─ layouts/             # 大屏布局组件
├─ views/               # 页面视图
├─ charts/              # ECharts 图表组件
├─ services/            # 数据服务层
├─ mock/                # Mock 数据
├─ stores/              # Pinia 状态管理
├─ types/               # TypeScript 类型定义
├─ utils/               # 工具函数和日志工具
├─ hooks/               # Vue 组合式函数
├─ config/              # 全局配置
├─ styles/              # 全局样式、变量、重置样式
└─ tests/               # 测试代码

四、页面效果要求

请实现一个漂亮、现代、专业的数据大屏首页。

页面主题：数境中枢

页面必须包含：

1. 顶部标题栏
   - 中文标题：数境中枢
   - 英文标题：DataSphereCenter
   - 当前日期时间
   - 系统运行状态

2. 核心指标区
   至少包含 4 个指标卡片：
   - 今日访问量
   - 活跃用户数
   - 数据接入量
   - 告警数量

3. 图表区
   至少包含以下模块：
   - 业务趋势折线图
   - 分类占比饼图或环形图
   - 区域排行柱状图
   - 实时告警列表
   - 系统资源使用率仪表盘、环形图或进度环

4. 视觉风格
   - 科技感
   - 深色背景
   - 蓝色、青色、绿色作为辅助色
   - 适合 1920x1080 大屏展示
   - 响应式适配常见浏览器窗口
   - 不要做成普通后台管理页面
   - 不要做成营销落地页
   - 不要用大段说明文字占据页面

五、Mock/API 切换要求

必须封装统一的数据访问层。

页面组件不能直接 import Mock 数据。

请设计以下数据流：

页面组件
 -> services
 -> dataProvider
 -> mockProvider 或 apiProvider

通过环境变量切换数据来源：

VITE_DATA_SOURCE=mock
VITE_DATA_SOURCE=api

当前默认使用 mock。

要求：

1. mockProvider 返回本地模拟数据。
2. apiProvider 使用 axios 请求真实接口。
3. services 对页面暴露统一方法。
4. 页面组件不关心数据来自 Mock 还是真实 API。
5. 后期切换真实 API 时，页面组件不用修改。

六、日志系统要求

请封装 logger 工具。

要求：

1. 支持 debug、info、warn、error。
2. 开发环境允许输出到 console。
3. 生产环境默认关闭 debug。
4. 预留后期接入远程日志上报的位置。
5. 页面关键操作和数据加载失败需要使用 logger。

七、测试系统要求

请配置并编写基础测试。

要求：

1. 使用 Vitest 做单元测试。
2. 至少为工具函数或数据服务写一个单元测试。
3. 使用 Playwright 做端到端测试。
4. 至少测试首页能正常打开，并且能看到“数境中枢”标题。
5. 在 package.json 中提供测试命令：
   - test
   - test:unit
   - test:e2e

八、代码质量系统要求

请配置：

1. ESLint
2. Prettier
3. Stylelint
4. TypeScript 类型检查

要求 package.json 中包含以下命令：

- dev
- build
- type-check
- lint
- lint:eslint
- lint:style
- format
- test
- test:unit
- test:e2e

九、工程规范要求

请遵循以下规范：

1. 使用 TypeScript。
2. 使用模块化开发。
3. 组件职责单一。
4. 页面只负责组合布局。
5. 图表组件放入 charts 目录。
6. 数据请求逻辑放入 services 目录。
7. Mock 数据放入 mock 目录。
8. 类型定义放入 types 目录。
9. 公共工具放入 utils 目录。
10. 全局样式放入 styles 目录。
11. 禁止把大量逻辑写在 App.vue 或 main.ts 中。
12. 禁止把所有图表写在一个巨大组件里。
13. 每个图表组件应该独立维护。
14. 代码要清晰，适合新手继续学习和维护。
15. 不要随意删除已有 README、LICENSE、Git 配置等项目基础文件。

十、README 文档要求

请更新 README.md，内容至少包含：

1. 项目介绍
2. 技术栈
3. 目录结构
4. 本地启动方式
5. 常用命令
6. Mock/API 数据切换方式
7. 测试说明
8. 代码质量说明
9. 开源协议说明

十一、验收要求

完成后请执行：

1. 安装依赖
2. 运行类型检查
3. 运行代码检查
4. 运行单元测试
5. 运行端到端测试
6. 构建项目
7. 启动开发服务器
8. 自动打开浏览器访问项目首页

如果某个步骤失败，请自行分析原因并修复，直到项目可以正常运行。

最终浏览器中应该能看到一个完整、美观、科技感的数据可视化大屏页面，标题为：

数境中枢
DataSphereCenter

十二、最终回复要求

完成后请告诉我：

1. 项目已完成哪些内容。
2. 本地访问地址。
3. 已执行并通过哪些检查。
4. 是否存在非阻塞提示，例如构建体积提示。
5. 后续建议我学习或开发的下一步。
```
